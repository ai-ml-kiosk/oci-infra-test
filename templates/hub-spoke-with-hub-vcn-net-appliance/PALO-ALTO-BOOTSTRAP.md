# Palo Alto Networks VM-Series Firewall Bootstrap

This document explains how to bootstrap the Palo Alto Networks VM-Series nodes deployed by the Core Landing Zone network appliance template. Core Landing Zone deploys the OCI infrastructure, the VM-Series instances, and the network load balancers. The firewall nodes still need an initial PAN-OS configuration before traffic can reliably pass through the appliance pair.

Run this procedure after the second Terraform or Resource Manager apply. Repeat the node-specific steps for both Palo Alto nodes.

## What the Bootstrap Configuration Does

The bootstrap configuration is defined in a XML file that is generated by [generate_panf_config.py](./generate_panf_config.py) script. It configures the following:

- Assigns the outdoor, or untrust, IP address to *ethernet1/1*.
- Assigns the indoor, or trust, IP address to *ethernet1/2*.
- Creates the *Untrust* and *Trust* zones.
- Creates static routes for default, external load balancer, and RFC 1918 private traffic.
- Enables the interface management profile used by load balancer health checks (TCP on port 22).
- Adds a basic outbound NAT rule for trust-to-untrust internet access.
- Adds a broad allow rule so the landing zone can become reachable before administrators replace it with production policy.

**The generated security policy is intentionally permissive. Treat it as a bootstrap baseline only. Firewall administrators should replace it with the required production security, NAT, logging, and threat-prevention policy after connectivity is validated.**

## Prerequisites

- The Palo Alto network appliance Terraform deployment has completed.
- You have the private key matching *net_appliance_public_rsa_key*.
- You can identify each node's management, outdoor, and indoor private IP addresses from the OCI Console.
- Python 3.10 or newer is available on the workstation or Cloud Shell session where you run the generator.
- For web UI access, you can create an OCI Bastion port-forwarding session to the Palo Alto management private IP on TCP port 443.

The default Hub VCN CIDR in *main.tf.paloalto.template* is *192.168.0.0/26*. Unless overridden, Core Landing Zone derives these Hub subnets from that CIDR:

| Subnet | Default CIDR | Purpose |
|--------|--------------|---------|
| Web subnet | *192.168.0.0/29* | External load balancer subnet |
| Outdoor subnet | *192.168.0.8/29* | Untrust firewall interface and outdoor NLB |
| Indoor subnet | *192.168.0.16/29* | Trust firewall interface and indoor NLB |
| Management subnet | *192.168.0.24/29* | Palo Alto management interface |
| Jump host subnet | *192.168.0.32/29* | Optional jump host and Core LZ Bastion service |

If you override Hub subnet CIDRs, use the actual subnet CIDRs from the OCI Console or Terraform state.

## Generate the Bootstrap XML

Use [generate_panf_config.py](./generate_panf_config.py) to generate one XML file per Palo Alto node. The script validates that the trust and untrust interface IPs are inside the supplied subnets, calculates the OCI subnet router IPs, and verifies that the generated output is valid XML.

From this template directory:

```bash
python generate_panf_config.py \
  --output panf-bootstrap.xml \
  --untrust-ip 192.168.0.14/29 \
  --trust-ip 192.168.0.22/29 \
  --untrust-subnet 192.168.0.8/29 \
  --trust-subnet 192.168.0.16/29 \
  --external-lb-subnet 192.168.0.0/29
```

Replace the example values as follows:

- *--untrust-ip*: private IP and prefix length assigned to the node's outdoor VNIC, mapped to PAN-OS *ethernet1/1*.
- *--trust-ip*: private IP and prefix length assigned to the node's indoor VNIC, mapped to PAN-OS *ethernet1/2*.
- *--untrust-subnet*: Hub outdoor subnet CIDR.
- *--trust-subnet*: Hub indoor subnet CIDR.
- *--external-lb-subnet*: Hub web subnet CIDR.

Generate a second file for the second node using that node's outdoor and indoor private IP addresses. The subnet CIDRs usually stay the same for both nodes.

## Initialize Each Firewall Node

Connect to the node management interface over SSH. Use one of the access patterns in [Connect to the Management Interface](#connect-to-the-management-interface).

Set the admin password:

```text
configure
set mgt-config users admin password
exit
exit
```

Enable jumbo frames:

```text
set system setting jumbo-frame on
```

Confirm the prompt when PAN-OS asks for confirmation.

## Upload the XML

You can upload the generated XML from the PAN-OS web UI or from the CLI. After either upload option, load and commit the configuration by following [Load and Commit the Configuration](#load-and-commit-the-configuration).

### Option 1: Web UI

1. Create an OCI Bastion port-forwarding session to the node management private IP on TCP port 443.
2. Open *https://localhost:<local-port>* in your browser.
3. Sign in as *admin* with the password configured in the previous step.
4. Go to **Device**, then **Setup**, then **Operations**.
5. Select **Import named configuration snapshot** and upload the generated XML file for this node.

### Option 2: CLI and SCP

Enable the SCP server on the firewall:

```text
configure
set mgt-config users admin preferences enable-scp-server yes
exit
exit
```

Copy the generated XML to the firewall from a host that can reach the management private IP. If you use the Core LZ jump host, prefer SSH agent forwarding instead of leaving private keys on the jump host.

```bash
scp -i <palo-node-private-key> panf-bootstrap.xml admin@<palo-node-mgmt-ip>:/scp/config/panf-bootstrap.xml
```

## Load and Commit the Configuration

This step is required after either upload option.

Load and commit the configuration:

```text
configure
load config partial mode merge from-xpath /config/devices to-xpath /config/devices from panf-bootstrap.xml
commit
exit
request restart system
```

Confirm the restart when prompted.

## Connect to the Management Interface

Palo Alto management interfaces are deployed in a private Hub VCN management subnet. Do not expose them with public IP addresses. Use one of the following access paths.

### SSH Through the Core LZ Jump Host

Use this path for initial CLI access.

1. Enable the jump host and Core LZ Bastion service during deployment:
   - *add_hub_vcn_jumphost_subnet = true*
   - *deploy_bastion_jump_host = true*
   - *deploy_bastion_service = true*
   - *bastion_service_allowed_cidrs = ["<your-workstation-public-cidr>"]*
2. In OCI Console, create a Bastion managed SSH session to the Core LZ jump host.
3. Copy and run the generated SSH command. Add agent forwarding if your security policy allows it.
4. From the jump host, connect to the Palo Alto management private IP:

```bash
ssh admin@<palo-node-mgmt-ip>
```

If you do not use agent forwarding, use an approved method to provide the Palo Alto private key to the jump host and remove it after the bootstrap session.

### Web UI Through an OCI Bastion Port-Forwarding Session

Use this path for PAN-OS web UI access. Core Landing Zone does not create this session for you.

Create an OCI Bastion port-forwarding session that targets the Palo Alto management private IP on TCP port 443. In deployments that follow the Core LZ remote-access model, create the Bastion service for this direct management path in the Hub VCN management subnet, or otherwise ensure the Bastion service subnet can route to the management interface and the management NSG allows the connection.

Using the OCI CLI:

```bash
oci bastion session create-port-forwarding \
  --bastion-id <bastion-ocid> \
  --target-private-ip <palo-node-mgmt-ip> \
  --target-port 443 \
  --ssh-public-key-file ~/.ssh/id_rsa.pub \
  --session-ttl 10800
```

When the session is active, copy the SSH command from the OCI Console session details, choose a local port such as *8443*, and run it. The command has this shape:

```bash
ssh -i ~/.ssh/id_rsa -N -L 8443:<palo-node-mgmt-ip>:443 -p 22 <bastion-session-ocid>@host.bastion.<region>.oci.oraclecloud.com
```

Keep the tunnel running, then open:

```text
https://localhost:8443
```
